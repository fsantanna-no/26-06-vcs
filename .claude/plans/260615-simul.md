# Reimplement TPD-21 Simulations on freechains.vcs (TCP)

## Context

The `tpd-21` paper (Sec. consensus.eval) simulates chat/newsgroup
forums running over Freechains to show practicability.
Original sims live in `old/tpd-21/{chat,usenet}/` (this project) and
target the old Kotlin daemon (`freechains-host --port`, TCP `peer`

`/x/freechains/vcs/`, keeping the network model **TCP** and staying
as close as possible to the original driver.

## How the vcs TCP model works (verified in source)

- `freechains --root=R daemon --port=P --hub`
    - runs `git daemon --base-path=R/chains/ --export-all`
    - `--hub` adds `--enable=receive-pack` (accept pushes)
    - upload-pack stays on by default (accept fetches)
    - so one `--hub` daemon serves BOTH send and recv
    - src: `src/freechains.lua:215`
- `chain '#c' sync recv <url>`  = `git fetch` (pull)
    - src: `src/freechains/chain/sync.lua` (recv branch)
- `chain '#c' sync send <url>`  = `git push -o freechains=true`
    - remote `pre-receive` hook re-runs `sync recv` then rejects
      the push with "Freechains: OK" (expected)
    - hook auto-installed on `chains add` init+clone
      (`src/freechains/chains.lua:107,159`)
    - src: `src/freechains/hooks/pre-receive`
- URL builder turns `localhost:P` into `git://localhost:P/<alias>`
    - default PORT 8330; src: `src/freechains/common.lua:78` (URL)
- Identity = the SSH pubkey string (80 chars, `ssh-ed25519 <b64>`)
  extracted from the signed commit, NOT a file path.
    - so the same keyfile yields the same author across peers
    - src: `src/freechains/chain/ssh.lua` (M.pubkey / M.verify)
- Virtual time = `--now=<sec>` per command (`CMD.now`), which sets
  `GIT_AUTHOR_DATE`/`GIT_COMMITTER_DATE` and persists into `G.now`.
    - no daemon-held clock; monotonic time travels with commits
    - reps consolidation scans fire when `time > G.now`
    - src: `src/freechains.lua:205`, `chain/common.lua` (apply)

## Old CLI -> vcs CLI mapping (TCP)

| original (common.lua)              | vcs (TCP)                                              |
|------------------------------------|-------------------------------------------------------|
| `freechains-host --port=P start D` | `freechains --root=D daemon --port=P --hub &`         |
| `freechains-host --port=P stop`    | `pkill -f "git daemon.*--port=P"`                     |
| `now <ms>` (daemon clock)          | `--now=<sec>` on each post/like (no separate step)    |
| `keys pubpvt 'pass'`               | `ssh-keygen -t ed25519 -N '' -C '' -f /tmp/u/<user>`  |
| `chains join '$chat' SHA` pioneer  | `chains add '#chat' init inline --sign=<pioneerkey>`  |
| `chains join '$chat' SHA` others   | `chains add '#chat' clone git://localhost:PIO/#chat`  |
| `chain c post M P --sign=PVT`      | `--now=ts chain '#c' post inline P --sign=<keyfile>` (add `--beg` if 0-rep) |
| `chain c heads blocked`            | `chain '#c' list begs`                                |
| `chain c like HASH --sign=PVT`     | `--now=ts chain '#c' like <N> post HASH --sign=<keyfile>` |
| `chain c consensus`                | `chain '#c' list order`                               |
| `peer localhost:P send c` (push)   | `--root=SRC chain '#c' sync send git://localhost:P`   |
| `peer localhost:P recv c` (pull)   | `--root=SRC chain '#c' sync recv git://localhost:P`   |

Notes:
- `like`/`dislike` args = `<number> <target> <id>` (number is a
  positive int; target = `post`|`author`; src `freechains.lua:179`).
  So unblock = `like 1 post <hash>`.
- `reps` args = `<target> [key]`; targets `post|posts|author|authors`
  (src `chain/reps.lua`). Author key trimmed of whitespace.
- Author identity for "like author" needs the 80-char pubkey
  string, obtainable via `ssh-keygen -y -f <keyfile>`.
  `cat <keyfile>.pub` matches the key stored from a signed commit
  (both are the same ed25519 wire format -> identical base64).
- "Welcoming like on a blocked post" (the old unknown) = liking a
  `beg`: `list begs` yields parked post hashes on `refs/begs/`;
  `chain like 1 post <hash>` merges/unblocks it
  (`src/freechains/chain/like.lua`, `list.lua` begs branch).

## Beg/unblock mechanism (verified in source)

Key shift from the old system: a 0-rep plain post no longer
auto-blocks — it **errors** ("insufficient reputation"). The poster
must opt in with `--beg`, so the harness decides up front.

| step    | old Kotlin        | vcs (Lua/Git)                              |
|---------|-------------------|--------------------------------------------|
| 0-rep   | `post` -> BLOCKED | `post --beg --sign` -> commit parked on     |
| post    |                   | `refs/begs/beg-<hash>`, main HEAD reset     |
|         |                   | (`post.lua` beg block, `reset --hard HEAD~2`) |
| detect  | `heads blocked`   | `git rev-parse refs/begs/beg-<hash>` /      |
|         |                   | `chain list begs`                           |
| unblock | pioneer `like`    | pioneer `like 1 post <hash>` -> `like.lua`  |
|         | `<hash>`          | detects beg, merges branch into main,       |
|         |                   | transfers reps, deletes the ref             |

Harness decision logic (replaces old post + `heads blocked`):

```
reps = `chain '#c' reps author <PUB>`
if reps <= 0:  post --beg --sign=key ; pioneer like 1 post <hash>
else:          post --sign=key
```

Note `<= 0`, not `== 0`: reps can go negative after dislikes; the
gate in `apply` is `reps <= 0`. Item (d) "extra likes" accounting
stays harness-side (as in old `common.lua`) — no CLI query needed.

## Spike: confirm beg/unblock before porting

Single root, mocked time; pass = `blocked_ok`, `unblocked_ok`,
`ref_clean`, and reps shifted pioneer -> newbie.

```sh
ROOT=/tmp/spike; rm -rf $ROOT
ssh-keygen -t ed25519 -N '' -C '' -f /tmp/pio
ssh-keygen -t ed25519 -N '' -C '' -f /tmp/new
FC="freechains --root=$ROOT"
REPO="$ROOT/chains/#spike"

$FC --now=0   chains add '#spike' init inline --sign=/tmp/pio
H=$($FC --now=100 chain '#spike' post inline 'hi im new' --beg --sign=/tmp/new)

git -C "$REPO" rev-parse --verify refs/begs/beg-$H            # exists => blocked
$FC chain '#spike' list order | grep -q $H && echo MAIN_BAD || echo blocked_ok

$FC --now=200 chain '#spike' like 1 post $H --sign=/tmp/pio   # welcome

$FC chain '#spike' list order | grep -q $H && echo unblocked_ok
git -C "$REPO" rev-parse --verify refs/begs/beg-$H 2>/dev/null && echo REF_LEAK || echo ref_clean
$FC --now=200 chain '#spike' reps author "$(cat /tmp/pio.pub)" # pioneer -1
$FC --now=200 chain '#spike' reps author "$(cat /tmp/new.pub)" # newbie +~1
```

Pass `--now=200` to the `reps` checks too: without it `CMD.now`
defaults to real time (~2026 >> 200), firing consolidation scans
(`time > G.now`) that inflate the newbie's reps. Direction holds
either way, but `--now` keeps the numbers clean.

## Multi-peer TCP gossip (chat-03 shape)

- pioneer peer creates `#chat`; all other peers `clone` from it
- per message:
    1. map author -> a fixed peer port (sticky)
    2. post signed by that user with `--now=ts`
    3. random `NSYNCS` peers: bidirectional `sync send` both ways
    4. if poster still `begs`, pioneer likes to unblock
- this reproduces forks/blocked metrics (paper items c, d)

- Beg/unblock (old high risk) RESOLVED by source read: `post --beg`
  + `like 1 post <hash>` (pending empirical spike run).
- Reps constants changed since the paper -> absolute numbers
  diverge; report methodology match, not figure match.
- `git daemon` receive-pack must run hooks; confirm push path
  end-to-end over `git://` (tests only cover local-path send).

  original one-host-per-port model).
- Monotonic `--now`: archives are time-sorted, so ts is
  non-decreasing (post requires `time >= G.now - diff`).
